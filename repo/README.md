# Field Service Membership & Data Governance System

A fully offline, role-based web application for managing field service
memberships, dispatch eligibility, searchable records, auditable activity, SLA
monitoring, and versioned CSV data cleansing. Built for secure single-site
deployment with no internet dependency.

## Architecture & Tech Stack

* **Frontend:** HTMX 1.9, Jinja2 templates, custom CSS (bundled for offline use)
* **Backend:** Python 3.11, Flask 3.0, Flask-Login, Flask-WTF, SQLAlchemy 2.0
* **Database:** SQLite (FTS5 full-text search; SQLCipher at-rest encryption in production)
* **Containerization:** Docker & Docker Compose (Required)

## Prerequisites

To ensure a consistent environment, this project is designed to run entirely within containers. You must have the following installed:
* [Docker](https://docs.docker.com/get-docker/)
* [Docker Compose](https://docs.docker.com/compose/install/)

## Running the Application

1. **Build and Start Containers:**
   Use Docker Compose to build the images and spin up the entire stack in detached mode.
   ```bash
   docker-compose up --build -d
   ```

   Note: the container generates encryption keys into `instance/` on first start. No `.env` file is required for default operation; to override secrets (e.g. `SECRET_KEY`, `SQLCIPHER_ENABLED`), pass them through the `environment:` block in `docker-compose.yml`.

2. **Seed the Database (first run only):**
   ```bash
   docker exec fieldservice-app python seed.py
   ```

3. **Access the App:**
   * Application: `http://localhost:5000`
   * Health Check: `http://localhost:5000/health`

4. **Stop the Application:**
   ```bash
   docker-compose down -v
   ```

## Testing

All unit, integration, frontend, and E2E tests are executed via a single, standardized shell script. This script automatically handles any necessary container orchestration for the test environment.

Make sure the script is executable, then run it:

```bash
chmod +x run_tests.sh
./run_tests.sh
```
*Note: The `run_tests.sh` script outputs a standard exit code (`0` for success, non-zero for failure) to integrate smoothly with CI/CD validators. Coverage threshold is enforced at `>= 90%`.*

| Suite | Count | Description |
| :--- | :--- | :--- |
| Unit | 75 | Service-layer logic (auth, members, workflow, eligibility, search, audit, SLA, cleansing) |
| Integration | 131 | HTTP routes, DB state, RBAC matrix, failure paths |
| Frontend | 16 | HTMX partials, form submissions, live updates |
| E2E | 6 | Full cross-module workflows with audit/timeline/SLA/eligibility verification |
| **Total** | **228** | **90% code coverage enforced** |
